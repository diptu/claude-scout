# Docusaurus Documentation Site

Helps set up and maintain a documentation website built with Docusaurus, a static-site generator purpose-built for versioned, searchable project documentation written in Markdown/MDX.

## When to apply this skill

- The user asks to create a new documentation site for a project (open-source or internal).
- The user already has a Docusaurus site and wants to add docs, versioning, a blog, or custom pages/components.
- The user wants to convert existing scattered docs (README, wiki, plain Markdown files) into a structured, navigable documentation site.
- The user asks about sidebar configuration, docs versioning, search integration, or theming for a Markdown-based doc site and Docusaurus is already in use or a reasonable fit.

Do not apply this skill for general static-site generation unrelated to documentation (e.g., marketing landing pages, blogs as the primary purpose) unless the user specifically wants Docusaurus.

## Core concepts to keep in mind

- **Content lives in Markdown/MDX**: `.md` files are plain Markdown; `.mdx` files allow embedding JSX/React components directly in the content.
- **Three content plugins by default**: `docs` (versioned, sidebar-driven reference material), `blog` (dated posts), and `pages` (standalone custom pages). Most projects only need `docs`.
- **Sidebar structure** is controlled by `sidebars.js`/`sidebars.ts` (or auto-generated from folder structure) — it defines the left-nav hierarchy independent of file layout.
- **Front matter** on each doc file (`id`, `title`, `sidebar_position`, `slug`, `description`) controls metadata, ordering, and routing.
- **Versioning** snapshots the current `docs/` folder into `versioned_docs/version-X.Y.Z/` and `versioned_sidebars/`, letting the site serve multiple doc versions side by side.
- **Site-wide config** lives in `docusaurus.config.js`/`.ts`: title, tagline, URL, base path, navbar, footer, plugins, themes, and deployment target.

## Step-by-step guidance

1. **Clarify scope first.** Ask (or infer from context) whether this is a brand-new site or additions to an existing one, whether versioning is needed, and where it will be hosted (GitHub Pages, Vercel, Netlify, custom).

2. **New site setup**:
   - Recommend `npx create-docusaurus@latest <site-name> classic --typescript` (or without `--typescript` if the project is plain JS) as the starting point — the `classic` template ships docs, blog, and pages preconfigured.
   - Walk through `docusaurus.config.js`: set `title`, `tagline`, `url`, `baseUrl`, `organizationName`, `projectName`, and the `presets` block pointing `docs`/`blog` at the right source directories.
   - If the project doesn't need a blog, disable it (`blog: false` in the preset config) rather than leaving unused boilerplate.

3. **Organizing content**:
   - Group related docs into subfolders under `docs/`; use `_category_.json` (or front matter `sidebar_label`/`sidebar_position`) to control how folders and files appear in the sidebar.
   - Prefer auto-generated sidebars (`sidebars.js` exporting `{tutorialSidebar: [{type: 'autogenerated', dirName: '.'}]}`) for simple projects; switch to a hand-written sidebar array only once the structure needs custom grouping or ordering that folder structure can't express.
   - Use MDX only when a page genuinely needs interactive/React content (tabs, live code blocks, custom admonitions) — plain `.md` is simpler and sufficient for most reference pages.

4. **Migrating existing docs**:
   - Convert existing Markdown files by adding front matter and fixing relative links to match Docusaurus's routing (links between docs use relative paths without the `.md` extension being required, but internal anchors and image paths should be verified).
   - Split a single large README/wiki page into multiple focused docs pages rather than dumping it as one giant page — this is what makes the sidebar and search useful.

5. **Versioning** (only when the user needs to support multiple released versions simultaneously):
   - Run the versioning command conceptually described as "snapshot current docs into a version" — explain that this freezes `docs/` content into `versioned_docs/version-<label>/` and creates a matching sidebar snapshot, and that future edits to `docs/` become the new "next" version.
   - Advise against introducing versioning prematurely — a project with one active release doesn't need it, and it adds real maintenance overhead (every doc fix must be back-ported to affected versions).

6. **Search**: for local/internal search, the built-in local search plugin (no external service) is the simplest option; for larger public sites, third-party indexed search (e.g. Algolia DocSearch) gives better relevance but requires external account setup — recommend based on site size and whether it's public.

7. **Deployment**: point the user to whichever static host fits their existing workflow (GitHub Pages via the `docusaurus deploy` command and `gh-pages` branch, or any static host by serving the `build/` output from `docusaurus build`). Make sure `url` and `baseUrl` in the config match the actual deploy target, since a mismatch is the most common cause of broken asset paths after deploy.

8. **Sanity check before finishing**: confirm the sidebar renders the intended hierarchy, internal links resolve, and (if applicable) the versioned dropdown shows the expected versions — mismatched `baseUrl`/`url` and broken relative links are the most common sources of a broken-looking site.
